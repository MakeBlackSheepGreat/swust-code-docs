# SWUST 优势

这页说明龙山灵码和 MiMo-Code 的关系，以及 SWUST 层实际增加了哪些行为。MiMo 已有能力按 MiMo 基座描述。

## 问题

真实仓库里的 AI 编程任务通常不是一次回答结束。常见问题包括：

- 上下文变长后，模型开始丢失早期约束
- 会话中断后，下一轮需要重新解释项目状态
- 子任务可以拆出去，但不同角色缺少项目级约束
- shell、写文件、结构化文档修改等操作带来工程风险
- 中文用户在终端里需要更直接的命令、状态和侧边栏信息

MiMo-Code 已经解决了其中的大部分运行时基础问题。龙山灵码的策略是保留这个基座，再补 SWUST 层。

## MiMo-Code 提供的基座

当前主线继承 MiMo-Code 的长任务运行时：

| 能力 | 作用 |
|------|------|
| memory / checkpoint | 保存项目事实、会话状态和恢复上下文 |
| actor / subagent | 把任务交给独立角色执行，并把结果带回主会话 |
| `goal` | 用停止条件检查任务是否真正完成 |
| `compose` | 组织规划、并行执行、复核和结果整合 |
| Dream / Distill | 把近期工作整理为长期知识或可复用流程 |
| MCP / LSP / plugin / skill | 接入外部工具、语言服务和项目扩展 |

这些能力在文档中应按 MiMo 基座能力描述。SWUST 不重写这些机制，也不把主线改回旧 OpenCode 风格。

## SWUST 层增加了什么

### 中文优先的终端信息

SWUST 层把终端内的高频信息改成更适合中文使用者阅读：

- 命令说明和常用入口使用中文表达
- 侧边栏突出工作目录、Goal、Task、Todo、LSP、MCP、变更文件、token、费用和缓存状态
- 首页、快速开始和功能页按“下一步做什么”组织，而不是按实现历史组织

这类改动的判断标准是使用者能否更快找到当前任务需要的信息。

### 工程防护

SWUST 层把几个容易造成事故的点单独处理：

| 模块 | 处理的问题 |
|------|------------|
| Task Gate | 仍有未完成任务时，减少过早停止 |
| Bash Safety | 对高风险 shell 命令做额外分析 |
| Write Guard | 收紧内存和项目写入路径边界 |
| Document Validation | 修改结构化文档时检查结构约束 |
| cache-stable prefix | 降低长任务中上下文前缀漂移带来的噪声 |

这些模块不替代 MiMo 的权限体系。它们补充的是更具体的工程边界。

### 项目级 subagent 配置

MiMo 已经提供 subagent 运行时。SWUST 额外增加项目级设置入口：

```text
/subagent
/subagents
```

可见子智能体可以单独设置：

- 模型
- 思考强度 / variant
- 最大执行步数
- 清除覆盖值，回到默认行为

配置写入项目级 `agent` 配置，而不是写入全局默认值。这样同一个项目可以把调查、实现、评审、验证等角色限制在不同模型和步数策略下。

### 记忆组织补充

SWUST 层保留 MiMo 的记忆和 checkpoint 机制，并补充更明确的项目知识组织方式：

- `MEMORY.md` 支持 `@path` 导入
- 项目事实可按 one-fact-per-file 保存
- 写入路径通过 guard 校验
- `/memory` 用于检索长期项目知识

这些设计的目标是减少“每次重新解释项目”的成本，同时避免记忆文件变成不可维护的长文本。

## 与普通 coding agent 的差别

普通 coding agent 往往把重点放在单轮生成质量。龙山灵码更关注任务跨时间推进时会发生什么。

| 问题 | 普通做法 | 龙山灵码的做法 |
|------|----------|----------------|
| 会话很长 | 压缩或丢弃历史 | 通过 memory、checkpoint 和上下文重建恢复必要状态 |
| 子任务很多 | 手动拆分 prompt | 使用 subagent / compose / goal 分担调查、执行、复核和完成判断 |
| 项目规则很多 | 反复提醒 | 写入项目记忆和事实文件，恢复时重新注入 |
| 操作有风险 | 依赖人工检查 | 用权限、Task Gate、Bash Safety、Write Guard 和 Document Validation 收紧边界 |

## 什么时候适合

更适合：

- 需要多轮推进的真实仓库任务
- 重构、迁移、升级、批量修复
- 需要子智能体分工的任务
- 需要跨会话保留项目规则和进度的任务

不特别适合：

- 一次性概念解释
- 没有仓库上下文的短问答
- 不需要记忆、checkpoint 或工具调用的临时问题

## 维护原则

后续开发按这个顺序判断：

1. MiMo-Code 已有的功能，优先保留 MiMo 当前实现。
2. MiMo-Code 没有、但 SWUST 任务需要的能力，再作为 SWUST 层加入。
3. Provider 和模型名称保持原名，例如 `MiMo Auto`、`mimo/mimo-auto`、`xiaomi/mimo-*`。
4. 公开文档只写稳定能力，不写临时分支、PR 状态或会话过程。
