# SWUST 优势

这页不是历史过程记录，也不是旧版 OpenCode 差异审计。它只回答一个更直接的问题：**在已经选择 MiMo-Code 作为基座的前提下，龙山灵码到底额外强化了什么，它适合解决哪些普通 AI coding agent 不擅长的问题。**

## 为什么以 MiMo-Code 为基座

龙山灵码当前主线之所以以 MiMo-Code 为基座，不是为了“换个名字继续做旧版本”，而是因为 MiMo 当前主线已经把长任务软件工程需要的关键运行时打通了：

- 持久化记忆与 checkpoint 不再是附加功能，而是会话连续性的核心部分
- actor / subagent、`goal`、`compose`、Dream / Distill、MCP、LSP、插件等能力位于同一运行时体系
- TUI、Server Runtime、Web / Desktop 入口共享一套底层能力
- 上下文重建、任务追踪、子智能体编排这些长任务关键路径已经形成体系

对龙山灵码来说，这样的基座更适合继续做产品层强化。它意味着 SWUST 不需要重复建设一套落后的运行时，而可以把精力放在中文体验、工程防护和项目治理上。

## SWUST 相对 MiMo 的增强层

MiMo 基座解决了“有没有完整运行时”的问题，SWUST 层主要解决“它在真实工程里是否更顺手、更稳、更适合中文开发者”的问题。

### 中文优先的 TUI 与信息组织

龙山灵码没有停留在品牌替换层面，而是把终端使用过程里最常见的信息重新组织为更适合中文阅读的结构：

- 关键命令、提示和面板文案优先保证中文可读性
- 侧边栏强调工作目录、Goal、Task、Todo、LSP、MCP、变更文件、token、费用、缓存等高频上下文
- 首页和主入口文档更强调“从哪里开始、下一步看什么”，减少首次使用的理解成本

这类改动的价值不在于看起来不同，而在于长时间使用时更少打断思路。

### 更完整的工程防护与安全治理

MiMo 已经具备权限体系，而 SWUST 进一步把注意力放在容易出工程事故的地方：

- Task Gate 防止 Agent 在仍有未完成任务时过早停止
- Bash Safety 对高风险命令做额外分析
- Write Guard 收紧写入路径边界
- Document Validation 面向结构化文档工作提供额外约束
- cache-stable 上下文布局降低长任务中上下文漂移带来的噪声

这些增强并不追求“看起来更强”，而是减少真实仓库里的误操作、误停和信息失真。

### 子智能体项目级个性化配置

MiMo 已经具备 subagent 运行时，而龙山灵码当前主线进一步补上了“项目级角色治理”这一层：

- 通过 `/subagent` 或 `/subagents` 打开可见子智能体设置
- 为不同子智能体分别设定模型
- 单独设定思考强度 / variant
- 单独设定最大执行步数

这意味着子智能体不只是“会被派出去工作”，而是可以被项目按角色约束成更具体的工程工具，例如轻量调查、重推理评审、短步执行器或受限验证器。

### 更偏长期项目的软件工程工作流

龙山灵码对记忆、checkpoint、Dream / Distill 的强调，不是为了多列几个高级词，而是为了形成长期项目闭环：

- 记忆用于保存长期规则和项目事实
- checkpoint 用于会话恢复和上下文重建
- Dream 用于整理近期工作里值得保留的项目知识
- Distill 用于把重复动作提炼为 skill、command、subagent 或工作流

这使得项目知识能够逐步沉淀，而不是每轮对话都重新解释。

## SWUST 相对普通 Coding Agent 的独特优势

相对普通的 AI coding agent，龙山灵码更强调以下三个问题。

### 1. 长任务如何不断线

很多 coding agent 在单轮任务里表现不错，但一旦任务跨越多个阶段、多个文件、多个会话，就容易出现：

- 忘记此前约束
- 重新理解项目
- 对未完成状态没有稳定感知
- 上下文膨胀后质量下降

龙山灵码把记忆、checkpoint、Goal、Task、subagent 和上下文重建放在同一条路径上，目的就是让任务在长链路里继续推进。

### 2. 多智能体如何真正服务工程

很多产品把“多智能体”理解为多开几个角色名称。龙山灵码更关心它们是否真的能承担工程分工：

- 主智能体负责主线推进
- compose 负责规划、并行、评审和合并
- goal 负责带停止条件的持续工作
- subagent 负责调查、实现、验证、checkpoint 等可拆分任务

再加上项目级子智能体配置后，多智能体不再只是展示性的“人设集合”，而是更接近可治理的角色系统。

### 3. 工程环境中的风险怎么收敛

真实工程环境里，风险往往不来自“模型答错一句话”，而来自：

- 不该执行的 shell 命令被执行
- 不该结束的任务提前结束
- 不该写的路径被写入
- 结构化文档被错误修改

龙山灵码的优势之一，是把这类问题当成一等问题处理，而不是留给使用者事后补救。

## 典型使用场景

| 场景 | 为什么适合龙山灵码 |
|------|--------------------|
| 长周期缺陷修复 | 需要 goal、task、checkpoint 和记忆共同保持连续推进 |
| 仓库升级改造与重构 | 需要跨多文件、多阶段、多角色协作 |
| 规范驱动开发 | 需要文档、架构规则、历史约束长期保留 |
| 大模型辅助评审 | 需要把不同子智能体限制在不同模型和步数策略下 |
| 重复工程流程沉淀 | 需要 dream / distill 持续把经验转化为可复用资产 |

## 适合与不适合的任务类型

### 更适合

- 真实代码仓库中的连续开发任务
- 需要恢复执行的长任务
- 需要显式分工、复核和回收结果的多智能体任务
- 需要长期积累项目知识的团队或个人项目

### 不特别适合

- 只追求一次性快速回答的浅层问答
- 只想把终端当作简化聊天窗口使用
- 不需要项目记忆、任务状态和恢复能力的临时脚本问答

## 一句话总结

如果把 MiMo-Code 视为“已经把长任务运行时打通的基座”，那么龙山灵码的定位就是：**在这个基座上，把中文优先体验、工程防护、子智能体治理和长期项目工作流做得更适合真实软件工程。**
