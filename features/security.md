# 安全防护

龙山灵码的安全模型不是单一开关，而是面向真实工程场景的多层治理。它既继承 MiMo 的权限运行时，也补上了 SWUST 更关注的工程防护层。

## 总体原则

当前主线的安全策略可以概括为三点：

- 高风险操作默认收紧，而不是默认放开
- 工程风险要在执行前被识别，而不是执行后再补救
- 长任务中的“误停、误写、误执行”都应被视为一等问题

这也是为什么龙山灵码会同时关心权限、Bash、安全写入、任务停止条件和文档结构。

## 防护层一览

| 防护层 | 主要作用 |
|--------|----------|
| 权限流水线 | 决定工具调用是允许、询问还是拒绝 |
| Bash Safety | 在 shell 执行前分析高风险命令模式 |
| Task Gate | 避免还有未完成任务时过早停止 |
| Write Guard | 收紧写入路径和记忆写入边界 |
| Document Validation | 对结构化文档工作增加额外约束 |
| 上下文治理 | 通过 cache-stable 布局减少长任务中的上下文漂移 |

其中，权限体系主要来自 MiMo 基座；Task Gate、Bash Safety、Write Guard、Document Validation 等更偏 SWUST 的工程增强。

## 权限流水线

工具调用不会直接“想跑就跑”，而是先经过权限判断：

1. 命中 blanket deny 规则时直接拒绝
2. 命中 blanket ask 规则时向用户确认
3. 交给工具自己的 `checkPermissions()` 做额外判断
4. 再由当前 mode 决定是否覆盖默认行为

这让“读文件”“改文件”“跑 shell”“访问外部资源”不会被混成同一类风险。

## Bash Safety

Shell 是工程效率的重要来源，也是风险最集中的地方之一。Bash Safety 会对命令做额外分析，重点拦截或提升警惕的场景包括：

- 危险删除
- 下载即执行
- 高风险权限修改
- 强制重置或强制推送
- 其他容易造成不可逆后果的操作

它的目标不是阻止一切命令，而是减少明显不该由 Agent 直接执行的那一类命令。

## Task Gate 与 Write Guard

这两层治理更偏向长任务的稳定性：

- Task Gate 关注“任务真的做完了吗”
- Write Guard 关注“这个位置真的该写吗”

前者防止模型在还有未完成工作时过早乐观收尾，后者则防止越界写入、错误记忆写入或不合适的目录修改。

## 为什么这对长任务更重要

短任务里，很多问题只会影响一轮回答；长任务里，问题会累积成状态错误：

- 一次误写会污染后续工作
- 一次误停会让任务链断掉
- 一次错误 shell 操作可能影响整个仓库
- 一次结构化文档误改会让后续自动化继续建立在错误基础上

龙山灵码之所以把这些能力纳入主线，不是为了“显得更严格”，而是因为它本来就服务于长期连续的软件工程任务。
